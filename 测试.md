# 为什么不考虑网络因素时性能仍然一般

不考虑网络因素，这个项目性能一般的核心原因不是 Netty 本身，而是 RPC 框架热路径上的固定成本比较高。

Demo 接口本身只是返回：

```text
Hello, name
```

业务几乎没有耗时，所以所有耗时都会集中暴露在框架本身：

```text
代理调用
服务发现
负载均衡
序列化
编解码
Future 匹配
线程切换
反射调用
```

## 1. 同步调用模型限制吞吐

底层虽然是 Netty 异步发送，但业务层默认还是同步等待：

```text
remoteCallAsync(...).get()
```

也就是每个调用线程发出请求后，会阻塞等待结果。

高并发时，很多业务线程处于等待 `CompletableFuture` 完成的状态。此时吞吐大致受这个公式影响：

```text
TPS = 并发数 / 单次调用平均耗时
```

所以当并发继续提升到 1500 后，TPS 不再明显增长，甚至下降，说明线程上下文切换、锁竞争和调度成本开始变高。

## 2. 每次请求对象创建较多

一次 RPC 调用会创建很多短生命周期对象：

```text
RpcRequest
MessageHeader
RpcMessage
RequestMetadata
RpcPromise / CompletableFuture
byte[]
ByteBuf
RpcResponse
List<ServiceInfo>
lambda / 回调对象
```

这些对象大多很快进入新生代，压测时会增加 Minor GC 压力。

Demo 接口越简单，框架对象创建成本占比越高。

## 3. 序列化器没有充分复用

当前 `SerializationFactory.getSerialization()` 每次都会创建新的序列化器：

```text
new JdkSerialization()
new JsonSerialization()
new HessianSerialization()
new KryoSerialization()
new ProtostuffSerialization()
```

这意味着每次编码、解码都可能重新创建序列化对象。

Protostuff 虽然在类里维护了 `LinkedBuffer`，但如果序列化器本身每次都 new，buffer 的复用效果就很有限。

Kryo 中虽然使用了 `ThreadLocal<Kryo>`，但每次序列化后又 remove，下一次仍然要重新创建 Kryo 对象，复用效果也会下降。

这部分是很大的性能损耗点。

## 4. 编解码存在额外内存拷贝

编码流程：

```text
对象 -> byte[] -> ByteBuf
```

解码流程：

```text
ByteBuf -> byte[] -> 对象
```

也就是每次都要申请 `byte[]`，并在 `ByteBuf` 和 `byte[]` 之间做拷贝。

成熟 RPC 框架会尽量减少中间数组拷贝，甚至直接基于 `ByteBuf` 做编解码。

## 5. 服务发现每次调用都有额外开销

虽然项目使用了 Curator `ServiceCache`，不是每次调用都访问 Zookeeper，但当前每次获取服务列表时仍然会从 `ServiceCache` 中读取实例，并重新构造服务列表。

后续还要做：

```text
过滤不可用 provider
判断健康状态
执行负载均衡
```

这些操作在高 QPS 下都会进入热点路径。

更理想的方式是：

```text
服务列表只在 cacheChanged 时更新。
RPC 调用时直接读取本地已经缓存好的不可变 provider 列表。
```

## 6. 服务端反射调用每次都查 Method

服务端处理请求时会根据方法名和参数类型获取方法：

```text
service.getClass().getMethod(...)
method.invoke(...)
```
这意味着每次调用都要查找 `Method`，然后再通过反射执行。

Demo 接口非常短，反射查找和调用成本就会非常明显。

可以优化为：

```text
serviceName + methodName + parameterTypes -> Method 缓存
```

进一步可以使用 `MethodHandle` 或 CGLIB FastClass 降低反射开销。

## 7. 业务线程池切换有成本

服务端收到 Netty 请求后，会把请求提交到业务线程池执行。

这个设计在真实业务场景下是正确的，因为不能让耗时业务阻塞 Netty EventLoop。

但在 Demo 压测里，业务方法太快，线程切换成本反而变明显：

```text
Netty EventLoop -> 业务线程池 -> 反射调用 -> 写回 Netty
```

同时默认业务线程池是 10 个线程，高并发下请求可能会排队，P99 容易升高。

## 8. 单 provider 单 Channel 可能形成局部瓶颈

客户端对每个 provider 复用一个 Channel。

优点是避免频繁建连。

但高并发下，所有请求的写入、响应读取和编解码都经过同一个 Channel 对应的 EventLoop，单连接可能成为局部瓶颈。

成熟框架通常会支持连接池，例如每个 provider 建多个连接，再按连接负载分摊请求。

## 9. 每个请求都会注册超时任务

每次发送请求时都会向 Netty EventLoop 注册一个超时任务。

如果 TPS 很高，并且超时时间设置较大，就会有大量定时任务堆在 EventLoop 的定时队列里。

即使请求已经很快返回，当前实现也没有主动取消 timeout task，而是等任务到期后再判断 Promise 是否已经完成。

这会增加 EventLoop 管理成本，也可能造成长尾延迟。

## 优化方向

最值得优先优化的点：

```text
1. 缓存序列化器、Schema、Kryo、Gson，减少重复创建。
2. 缓存服务端 Method，避免每次反射查找方法。
3. 请求完成后取消 timeout task，减少 EventLoop 定时任务堆积。
4. 服务发现列表只在节点变化时刷新，请求热路径直接读本地不可变缓存。
5. 支持每个 provider 多 Channel 连接池，降低单连接瓶颈。
6. 减少 ByteBuf 和 byte[] 之间的中间拷贝。
```

面试总结：

```text
我的压测接口是 Demo 接口，业务耗时几乎为 0，所以测出来的瓶颈主要集中在 RPC 框架自身，
包括序列化、编解码、请求响应匹配、线程切换和反射调用。
后续优化会优先从序列化器复用、方法缓存、减少对象分配、取消无效超时任务和连接池化入手。
```
